            What‚Äôs this MMU?

            Memory Management Unit (MMU) is the hardware inside our 64-bit Single-Board Computer (SBC) for‚Ä¶

            • Memory Protection: Prevent Applications (and Kernel) from meddling with things (in System Memory) that they‚Äôre not supposed to

            • Virtual Memory: Allow Applications to access chunks of ‚ÄúImaginary Memory‚Äù at Exotic Addresses (0x8000 0000!)‚Ä¶

              But they‚Äôre actually System RAM recycled from boring old addresses (like 0x5060 4000)

              (Kinda like ‚ÄúThe Matrix‚Äù)

            Sv39 sounds familiar‚Ä¶ Any relation to SVR4?

            Actually Sv39 is the MMU inside many RISC-V SBCs‚Ä¶

            • Pine64 Ox64, Sipeed M1s

              (Based on Bouffalo Lab BL808 SoC)

            • Pine64 Star64, StarFive VisionFive 2, Milk-V Mars

              (Based on StarFive JH7110 SoC)

            In this article, we walk through the steps to configure the Sv39 MMU on Pine64 Ox64 64-bit RISC-V SBC (pic below), powered by Bouffalo Lab BL808 SoC.

            We start with Memory Protection, then Virtual Memory. We‚Äôll do this with Apache NuttX RTOS. (Real-Time Operating System)

            And ‚ÄúSv39‚Äù means‚Ä¶

            • ‚ÄúSv‚Äù signifies it‚Äôs a RISC-V Extension for Supervisor-Mode Virtual Memory

            • ‚Äú39‚Äù because it supports 39 Bits for Virtual Addresses

            • Coincidentally it‚Äôs also 3 times 9: 9 Bits for Level 1, 9 Bits for Level 2, 9 Bits for Level 3!

            This is the simplest setup for Sv39 MMU that will protect the I/O Memory from 0x0 to 0x3FFF FFFF‚Ä¶

            All we need is a Level 1 Page Table (4,096 Bytes).

            The Page Table contains only one Page Table Entry (8 Bytes) that says‚Ä¶

            • V: This is a Valid Page Table Entry

            • G: This is a Global Mapping that is valid for all Address Spaces

            • R: Allow Reads for 0x0 to 0x3FFF FFFF

            • W: Allow Writes for 0x0 to 0x3FFF FFFF

            • PPN: Physical Page Number (44 Bits) is 0x0

              (PPN = Memory Address / 4,096)

            But we have so many questions‚Ä¶

            1. Why 0x3FFF FFFF?

              This is a Level 1 Page Table. Every Entry in the Page Table configures a (huge) 1 GB Chunk of Memory.

              (Or 0x4000 0000 bytes)

              Our Page Table Entry is at Index 0. Hence it configures the Memory Range for 0x0 to 0x3FFF FFFF. (Pic below)

            2. How to allocate the Page Table?

              In NuttX, we write this to allocate the Level 1 Page Table: jh7110_mm_init.c

              // Number of Page Table Entries (8 bytes per entry)
#define PGT_L1_SIZE (512)  // Page Table Size is 4 KB

// Allocate Level 1 Page Table from `.pgtables` section
size_t m_l1_pgtable[PGT_L1_SIZE]
  locate_data(".pgtables");

              .pgtables comes from the NuttX Linker Script: ld.script

              /* Page Tables (aligned to 4 KB boundary) */
.pgtables (NOLOAD) : ALIGN(0x1000) {
    *(.pgtables)
    . = ALIGN(4);
} > ksram

              Then GCC Linker helpfully allocates our Level 1 Page Table at RAM Address 0x5040 7000.

            3. What is SATP?

              SATP is the RISC-V System Register for Supervisor Address Translation and Protection.

              To enable the MMU, we set SATP Register to the Physical Page Number (PPN) of our Level 1 Page Table‚Ä¶

              PPN = Address / 4096
    = 0x50407000 / 4096
    = 0x50407

              This is how we set the SATP Register in NuttX: jh7110_mm_init.c

              // Set the SATP Register to the
// Physical Page Number of Level 1 Page Table
mmu_enable(
  g_kernel_pgt_pbase,  // 0x5040 7000 (Page Table Address)
  0  // Address Space ID
);

              (mmu_enable is defined here)

            4. How to set the Page Table Entry?

              To set the Level 1 Page Table Entry for 0x0 to 0x3FFF FFFF: jh7110_mm_init.c

              // Map the I/O Region in the MMU
mmu_ln_map_region(
  1,             // Level 1
  PGT_L1_VBASE,  // 0x5040 7000 (Page Table Address)
  MMU_IO_BASE,   // 0x0 (Physical Address)
  MMU_IO_BASE,   // 0x0 (Virtual Address)
  MMU_IO_SIZE,   // 0x4000 0000 (Size)
  MMU_IO_FLAGS   // Read + Write + Global
);

              (mmu_ln_map_region is defined here)

            5. Why is Virtual Address set to 0?

              Right now we‚Äôre doing Memory Protection for the Kernel, so we set‚Ä¶

              Virtual Address = Physical Address = Actual Address of System Memory

              Later when we configure Virtual Memory for the Applications, we‚Äôll see interesting values.
